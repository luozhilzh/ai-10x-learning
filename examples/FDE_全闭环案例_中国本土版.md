## 第 0 步 · 一句话先立住（中美 FDE 根本差异）

**美国 FDE** ≈ 把云上的通用模型/产品，现场调到客户肯付钱 → 卖的是「标准化产品 + 现场适配」。
**中国 FDE** ≈ 把大模型在「私有化/信创/等保/关系 + 招投标」的土壤里，落地成客户肯**签单、肯续费**的结果 → 卖的是「结果」，且多了一项美国同事几乎不用扛的硬技能：**合规与国产适配**。

> 一句话记忆：国外 FDE 是「把模型搬进客户业务」；国内 FDE 是「把模型 + 关系 + 合规一起搬进客户的院子，还得用客户家的铲子（信创）挖」。

## 第 1 步 · 五视角 STORM（中国 FDE 主题）

**1. 实践者（在岗国内 FDE，如腾讯云/字节豆包 FDE、驻场交付）**
- 立场：国内 FDE 不是"写代码或做销售"二选一，而是**当天在客户内网调私有化部署、当天跟信息化主任聊决策链、当天把坑喂回产品**的混合体；且「数据不出域」让模型效果天然打折，你得在合规壳子里把活干漂亮。
- 最强证据：真实项目里大量时间花在**信创适配（国产芯片/OS 跑不起来）、等保三级卡脖、招投标材料**——这些美国 FDE 几乎不碰。
- 只有他会说：最累的不是技术也不是关系，是**「一把手拍板、用的人不买账、信息化主任两头受气」的三角张力**，JD 永远不写。

**2. 学者（研究中国 ToB / AI 落地的人）**
- 立场：FDE 在中国不是"美国风口照搬"，而是**中国 SaaS 困境（信任稀缺、NDR<100%、定制化陷阱）的必然产物**——客户不信"工具"只信"结果"，所以必须派个人去把活干完。
- 最强证据：中美 SaaS 对比研究显示，美国头部 SaaS 的 NDR（净收入留存率）普遍 >120%，中国大部分 <100%；信任基础不同，决定了"卖结果的人"在中国更有存在价值。
- 只有他会说：国内产品成熟度参差，**FDE 既可能是高杠杆的"共创"，也可能是低成长的"实施"**，取决于公司是否真有标准化产品——别把 A 厂的 FDE 套到 B 厂。

**3. 怀疑者（吐槽者 / 过来人）**
- 立场：国内很多"FDE"只是**把传统实施工程师 / 技术售前换皮涨价**，甚至被"35 岁焦虑"营销裹挟——本质上是「高级集成苦力」。
- 最强证据：定制化的 FDE 最容易被锁死在每个客户重写一遍的泥潭，**模型红利不在你手里**（你部署的是别人/开源的模型），技术复利被稀释；且出差/驻场对生活侵蚀大。
- 只有他会说：真能"起飞"的是少数——多数人的杠杆来自**趁早进对一家产品正在起量的公司 + 主动把共性沉淀成标准资产**，不是"FDE"这三个字。

**4. 经济学家（看钱与利益）**
- 立场：国内云厂商（字节/腾讯/阿里/蚂蚁/智谱）抢 FDE，是因为**它能把"卖不出去的通用大模型"变成"客户肯签单、肯续费的具体结果"**，在关系型市场里抢企业 AI 入口的 ROI 极高。
- 最强证据：国内 FDE 月薪约 35–70K×13–15 薪（头部个案年薪破百万）；美国 OpenAI/Anthropic FDE 年薪约 17–20 万美元——国内绝对值低，但相对国内工程师溢价明显，且在关系型市场里"能搞定客户的人"议价权高。
- 只有他会说：FDE 的薪酬**低于同级别美国核心岗、但高于国内售前/实施**；所谓"离钱近"更多是职业期权（转产品/创业），不是当期现金。

**5. 历史学家（看先例）**
- 立场：FDE 不是新发明；美国是 Palantir 现场工程师的 AI 升级版，中国是**「驻场实施 / 系统集成 / 现场工程师」的 AI 升级版**，且每轮技术平台（主机→PC→云→AI）中国都晚半拍但会本土化（如信创、私有云）。
- 最强证据：IBM 现场系统工程师、军工 field engineer、传统 SI（系统集成商）的实施团队，都是"懂技术又贴客户"的先例；中国只不过把这套塞进了大模型 + 信创 + 等保的壳。
- 只有他会说：下一轮平台更替，FDE 会再换皮——**但"把新技术搬进中国企业"的现场角色永远在**，只是名字和合规包袱会变。

## 第 2 步 · 矛盾图谱

- **直接冲突的视角**
  - 学者（"FDE 在中国更有价值，因为卖结果"）↔ 怀疑者（"只是换皮的实施苦力"）：冲突在**"角色本身价值"还是"你落进的具体公司/部署形态"**——同样叫 FDE，私有化高度定制=填坑，标准化产品赋能=造火箭。
  - 经济学家（"关系型市场 FDE 溢价高"）↔ 怀疑者（"被锁死成苦力"）：两件事不矛盾——溢价只属于**"能拿反馈权 + 能搞定关系"的少数**，多数人拿不到。
- **证据强弱**
  - 最强：实践者的"信创/等保/数据不出域"、学者的"依赖产品成熟度"——可观察的真实规律。
  - 最弱："FDE=不写代码躺赢年薪百万"的营销话术——国内 FDE 代码量不低（私有化调试/脚本/集成），百万是头部个案。
- **一句能化解最大矛盾的问题**：*「这家公司的 FDE，是在'用私有化部署做高度定制的实施'，还是在'用相对标准化的产品做共创赋能'？」*——答清这个，学者与怀疑者的冲突就消解了。
- **所有视角都同意的（大概率是真的）**：① 中国 FDE 绕不开**私有化部署 / 信创 / 等保 / 数据合规**；② 关系 + 招投标文化让 discovery 重心从"用户痛点"转向"决策链 + 关键人"；③ 国内产品成熟度参差，FDE 易陷定制。
- **没人提到的盲区**：**退出机制与长期技能复利**——在定制化泥潭里 FDE 反而比美国更难转出"可迁移能力"，因为每家的信创/数据/关系都不可复用；能不能把共性问题沉淀成公司标准资产，决定你离开时"除出差经验外还剩什么"。

## 第 3 步 · 综合简报

**一句话（给只有 60 秒的 CEO）**：中国 FDE 是驻扎在客户现场、既调私有化部署又懂业务还扛关系与合规的"结果落地尖兵"——在"关系驱动 + 私有化 + 强合规 + 信创生态"的土壤里，把大模型从演示变成客户肯签单、肯续费的结果；它比美国 FDE 更离不开"卖结果"和"搞定人"，但也被子**定制化 + 合规**双重夹击，关键看公司给不给你标准化产品杠杆和反馈权。

**5 个关键发现（按可靠性排序）**
1. 【实践者+经济学家共识】中国 FDE **真实存在且高薪**：字节豆包 FDE 35–70K×15 薪、腾讯云/阿里云扩招、智谱 FDE 负责人 6–8 万/月。（可靠度 9，有多方 JD 与报道）
2. 【实践者+学者共识】**部署形态根本不同**：中国以私有化/一体机/信创适配为主，美国以云 API/SaaS 为主——合规与信创驱动，多来源一致。（可靠度 9）
3. 【实践者】**合规是第四项硬技能**：PIPL/数据安全法/等保三级/数据出境安全评估/国密/算法备案——美国 FDE 在国内基本不直面这套。（可靠度 9，法律文献充分）
4. 【学者+历史学家】**客户文化与销售机制不同**：关系驱动 + 招投标 + 一把手拍板 + 定制化陷阱，导致 discovery 重心从"用户痛点"转向"决策链 + 关键人"。（可靠度 8）
5. 【怀疑者】国内 FDE **易被"定制化 + 低产品成熟度"稀释成实施苦力**，杠杆取决于公司是否标准化。（可靠度 7，取决于具体公司）

**隐藏关联**：中国"信任稀缺 + NDR 低 + 定制化"的 SaaS 困境，恰好是 FDE **存在的根本原因**（卖结果而非工具）；但同一套困境也让 FDE **最容易掉进"为每个客户重写"的陷阱**——解法不是更努力定制，而是推动公司把共性问题做成标准化资产（这正是国内"反馈权"的真实含义，比美国更稀缺更值钱）。

**行动建议（以你的角色：想在国内做/转 FDE 的技术人）**
1. 补「私有化部署 + 信创适配 + 等保」技术栈——这是和美国 FDE 最大的能力差。
2. 练「决策链 mapping + 关系经营」而不仅是用户访谈。
3. 面试直接问"交付是私有化定制还是标准化产品赋能"判断杠杆。
4. 复用你已有的"非结构化抽取 → 行业知识库 → 固化产品模板 → 多家复用"故事（`FDE_面试故事_可面试版.md`），它天然契合国内"把个例沉淀成模板"的稀缺诉求。

**前沿问题**：国产模型（DeepSeek/Qwen）一体机普及后，部署门槛骤降，FDE 会从"部署苦力"升级为"业务编排 + Agent 工程"吗？还是被标准化一体机吞掉？答案决定这个角色在国内再活五年还是换皮。

## 第 4 步 · 同行评审自检（AI 给自己的简报挑刺）

- **逐条可靠性复评**：
  1. 真实存在且高薪 → 9（JD 与多方报道一致）。
  2. 部署形态不同 → 9（合规与信创驱动，多来源一致）。
  3. 合规是第四硬技能 → 9（法律文献充分）。
  4. 客户文化不同 → 8（一线共识，但"定制化陷阱"程度因公司而异）。
  5. 易被稀释 → 7（真实风险，但取决于个人主动性与公司产品力，非必然）。
- **最没把握的结论**：第 5 条"定制化稀释"的程度——不同公司（云厂商 vs AI 原生厂 vs 传统集成商）天差地别，需查具体 JD 与一线访谈验证。
- **比重过大的视角**：实践者视角（招商/JD 层面）可能高估岗位光鲜度；应加重"怀疑者 + 历史学家"对**长期职业**的判断。
- **该加的第 6 个视角**：**国内客户 CIO / 信息化负责人**——他会直接说"我们真正要 FDE 解决什么、为什么内部团队搞不定"，这能校准需求真实性，避免被厂商叙事带偏。
- **斯坦福教授若来评审**：会打 B−；会让我改：① 区分"岗位定义(新兴)"与"本土实现(高度不定)"，别混谈；② 补"如何衡量一个国内 FDE 成不成功"的可操作指标（如交付周期缩短率、代码采纳率、客户续约/增购）；③ 第 5 条要给出"怎样避免被定制化锁死"的具体动作（推动标准模板/资产沉淀）。

## 第 6 步 · 学习阶梯（5 级，从小白到自信实践者）

**L1 完全初学者**
- 理解什么：中国 FDE 是什么、和国内传统售前(SE)/实施工程师/解决方案架构师的根本区别。
- 掌握的样子：能用一句话向朋友解释"国内云厂商为什么都在抢 FDE"。
- 重点：角色四合一（编码+业务翻译+关系决策链+合规）、驻场属性、与美版差异。
- 里程碑：能说清"为什么国内比美国更需要'卖结果'的 FDE"。
- 练习：列出 3 个你眼中"国内 FDE"与"传统实施"的不同。
- 常犯错误：以为 FDE = 换皮销售或 = 纯外包；或以为照搬美国 Palantir 叙事即可。
- 自测：国内 FDE 相比售前(SE)最关键的差异？→（更贴产品、写更多生产级代码、把一线经验反哺成标准资产）

**L2 基本理解**
- 理解什么：一次**国内部署闭环**（发现决策链 → 私有化 PoC → 信创集成 → 等保上线 → 反馈产品）、所需技能组合（四项）。
- 掌握的样子：能画出 FDE 在"客户决策链—产品—合规"间的位置与双向反馈。
- 重点：闭环每步在做什么、四项技能权重（编码/业务翻译/关系决策链/合规）。
- 里程碑：能讲清"一次国企/金融客户 PoC 里 FDE 具体做哪几件事"。
- 练习：画一次国内部署闭环图。
- 常犯错误：只盯编码/模型，忽略"决策链"与"合规"两块国内特有短板。
- 自测：FDE 把一线痛点喂回产品，公司得到什么？→（缩短签单/交付周期、提高 PoC 与续费成功率、指导路线图）

**L3 实际使用者**
- 理解什么：如何在国内客户（国企/金融/制造）跑一次 discovery、做私有化原型、集成进客户内网（企微/钉钉/飞书）。
- 掌握的样子：能独立主持一次"含决策链 mapping"的客户发现，并产出数据不出域的 mini 原型。
- 重点：招投标语境下的需求发现、数据分级、国产生态集成。
- 里程碑：给一个"假客户（如某制造企业）"做完 discovery + 一个能本地跑的 RAG demo。
- 练习：挑熟悉场景，写 discovery 问题清单（含"谁拍板/数据能出域吗/用什么信创环境"）+ 原型。
- 常犯错误：一上来写大系统，而不是先验证最小假设；忽略"谁决策"导致做了白做。
- 自测：discovery 阶段最先该搞清楚什么？→（真实痛点、决策链与关键人、数据分级与出域限制、现有信创环境、约束）

**L4 问题解决者**
- 理解什么：处理信创适配失败、等保卡脖、stakeholder 政治（关系方扯皮）、范围蔓延（客户"再改一点"）。
- 掌握的样子：能在"无文档、客户防备、合规卡脖、关系复杂"下推进并设边界。
- 重点：信创/权限、合规红线、决策链经营与向上管理。
- 里程碑：能写出一个"项目边界 + 升级路径 + 关键人"的简短说明。
- 练习：针对 L3 原型，模拟客户临时加需求 + 等保拦路 + 信息化主任和业务的拉扯，写应对。

## 第 10 步 · 一页速查表（5 分钟扫完）

**一句话**：中国 FDE = 在"关系驱动 + 私有化 + 强合规 + 信创生态"土壤里，把大模型落地成客户肯签单、肯续费的结果的人；比美国 FDE 多扛"合规 + 关系决策链"两座山。

**中美 FDE 差异总表（本包核心）**
| 维度 | 国外（美/西）FDE | 中国本土 FDE |
|---|---|---|
| 部署形态 | 云 API / SaaS 为主 | 私有化 / 一体机 / 信创适配为主 |
| 销售/客户文化 | 产品驱动、ROI 演示、续费 KPI（NDR>120%） | 关系驱动、招投标、一把手拍板、定制化（NDR<100%） |
| 核心难点 | 把模型适配进客户业务流 | + 信创适配、等保、数据不出域、决策链经营 |
| 第四硬技能 | 一般无（GDPR/SOC2 由专门团队） | **合规与国产适配（PIPL/等保/数据出境/国密）** |
| 生态绑定 | 开放生态、互操作 | 封闭生态（企微/钉钉/飞书/微信）必须集成 |
| 薪酬量级 | OpenAI/Anthropic 约 17–20 万美元/年 | 头部 35–70K×13–15 薪，个案破百万 RMB |
| discovery 重心 | 用户痛点 + 业务价值 | **决策链 + 关键人 + 数据出域限制** |
| 杠杆来源 | 反馈权 + 需求→系统能力 | 反馈权（=推标准资产）+ 关系 + 合规落地能力 |

**核心条目**
- 本质：编码 + 业务翻译 + 关系决策链 + 合规，四合一，介于 SWE 与售前之间但更偏技术。
- 闭环：发现决策链 → 私有化 PoC → 信创集成 → 等保上线 → 反馈产品。
- 组织位置：桥接"客户决策链/合规"与"核心产品"，双向反馈是价值来源。
- 公司视角：缩短签单/交付周期、提高 PoC 与续费成功率，在关系型市场抢 AI 入口 ROI 高。
- 个人视角：杠杆来自"反馈权（推标准资产）+ 关系 + 合规落地能力"，是职业期权而非当期高薪。
- 关键风险：易被定制化 + 低产品成熟度稀释成出差型集成苦力；体验取决于部署形态与产品阶段。

**3–5 个真实场景**
1. 某国企/银行要上大模型 → FDE 驻场，在**等保三级 + 数据不出域**下用本地模型搭 RAG/风控原型并集成进内网。
2. 客户临时加需求 + 等保拦路 → FDE 设边界、判断升级，而非硬扛。
3. 多个客户都抱怨同一痛点（如某类文档抽不准）→ FDE 把共性反馈回产品，下个版本内置解决（**国内最稀缺的杠杆**）。
4. 信创环境（国产芯片/OS）跑不起来 → FDE 做适配、写最小依赖方案，而不是甩给客户。
5. 你只交付不积累 → 离开时除"出差 + 信创踩坑"别无长物（反面教材）。

**新手易错 / 易混**
- 以为 FDE = 高级销售 或 = 纯外包（国内版也别这么想）。
- 以为不用深抠代码也能混——能写生产级 + 私有化调试才是和售前的分水岭。
- 照搬美国 Palantir 叙事——忽略信创/合规/关系三座大山。
- 只看一家 JD 就下结论——云厂商 vs AI 原生厂 vs 传统集成商，差异巨大。
- 忽略"关系决策链 + 合规"这两块国内特有短板。

**上场前检查清单**
- [ ] 能一句话讲清中国 FDE 与国外 FDE 的 3 个根本不同
- [ ] 画得出国内部署闭环、说得出每步（含等保/数据出域）
- [ ] 给自己的 编码/业务翻译/关系决策链/合规 四项打了分、知道短板
- [ ] 按"部署形态 + 产品阶段 + 反馈权"筛好了目标公司清单
- [ ] 准备了一个"数据不出域"的本地 mini 原型（如本地 RAG demo）
- [ ] 想清"反馈权"怎么争取（推标准资产）、经历怎么转化成下一步
- [ ] 面试能反向问"交付是私有化定制还是标准化产品赋能"

**5 个快问快答（测你还记得多少）**
1. 中美 FDE 部署形态最大不同？→ 中国私有化/信创为主，美国云 API/SaaS 为主。
2. 中国 FDE 的"第四硬技能"是？→ 合规与国产适配（PIPL/等保/数据出境/国密）。
3. discovery 重心为何从"用户痛点"变"决策链"？→ 关系驱动+招投标+一把手拍板+定制化陷阱。
4. 什么情况该升级而非硬扛？→ 合规红线、超授权、产品核心缺口、招投标违规。
5. 为什么说"定制化"是双刃剑？→ 卖结果需要定制，但也最易锁死成实施苦力、攒不到可迁移能力。
